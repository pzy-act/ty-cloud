# TY Cloud 开发推进说明

更新时间：2026-05-02

## 业务主线

系统按“招生 -> 课程/班级 -> 排课 -> 上课/考勤 -> 收费 -> 复盘”推进。

核心模块：

1. 组织与权限：校区、用户、角色、菜单权限、登录鉴权。
2. 学员管理：线索、跟进、渠道、学员档案、家长、标签。
3. 课程与班级：课程、课程包、班级、课时价格。
4. 排课与资源：课表、教室、设备、资源预约。
5. 报名与合同：报名、协议、优惠、课包有效期。
6. 财务订单：订单、收款、退款、发票。
7. 考勤与课消：签到、请假、补课、课时扣减。
8. 教师与绩效：教师档案、提成规则、薪资核算。
9. 数据报表：招生转化、满班率、出勤率、续费率、营收成本。

## 鉴权流程

- 登录：`POST /api/system/auth/login`
- 刷新 Token：`POST /api/system/auth/refresh`
- 当前用户：`GET /api/system/auth/me`
- 退出：`POST /api/system/auth/logout`

前端请求业务接口时携带 `Authorization: Bearer {accessToken}`。后端通过 `JwtAuthenticationFilter` 校验令牌，并将用户信息写入 `SecurityContext`。

生产环境需通过环境变量 `TY_JWT_SECRET` 覆盖默认 JWT 密钥。

## 小程序端架构设计

用户端建议采用“小程序独立前端 + 后端共用一个 Spring Boot 项目”的架构。后台管理 Web 和小程序端访问同一套业务数据库，但接口、认证、权限边界分开，避免小程序直接复用后台管理接口。

### 项目拆分

推荐项目结构：

```text
ty-cloud          后端 API，一个 Spring Boot 项目
ty-cloud-web      后台管理 Web，一个 Vue 前端应用
ty-cloud-mini     小程序端，一个 uni-app / 微信小程序前端应用
```

部署形态：

- `ty-cloud-web`：后台管理端，面向管理员、教务、前台、财务、校区负责人，PC 浏览器访问为主。
- `ty-cloud-mini`：家长/学员/教师移动端，微信小程序访问为主。
- `ty-cloud`：统一后端服务，提供后台接口和小程序接口，共用实体、Mapper、Repository、数据库事务和核心业务规则。

### 后端模块边界

后端不另起项目，仍保留一个 `ty-cloud`。在同一项目中按接口场景分 Controller 和 Service：

```text
controller
├─ system          后台系统管理接口
├─ student         后台学员管理接口
├─ course          后台课程管理接口
├─ finance         后台财务管理接口
├─ attendance      后台考勤课消接口
├─ artwork         后台作品管理接口
└─ mini            小程序端接口
   ├─ MiniAuthController
   ├─ MiniStudentController
   ├─ MiniLessonController
   ├─ MiniBookingController
   ├─ MiniBalanceController
   ├─ MiniArtworkController
   └─ MiniFeedbackController

service
├─ student
├─ course
├─ finance
├─ attendance
├─ artwork
└─ mini
   ├─ MiniAuthService
   ├─ MiniStudentPortalService
   ├─ MiniBookingService
   └─ MiniBalanceService
```

原则：

- `entity`、`mapper`、`repository` 继续共用，避免重复建模。
- 后台接口保留 `/api/system/**`、`/api/student/**`、`/api/course/**` 等路径。
- 小程序接口统一放在 `/api/mini/**` 下。
- 小程序 Controller 只暴露家长/学员/教师需要的能力，不暴露后台管理型 CRUD。
- 复杂业务联动放在 Service 层，跨表动作使用 `@Transactional(rollbackFor = Exception.class)`。

### 认证与权限

后台管理端：

```text
账号密码登录 -> 后台 JWT -> 用户/角色/菜单权限
```

小程序端：

```text
微信登录 code -> 后端换取 openId -> 绑定手机号/家长账号 -> 小程序 JWT
```

权限隔离：

- 后台用户按角色、菜单、按钮权限控制。
- 小程序用户按微信身份、家长账号、绑定学员关系控制。
- 家长只能查看和操作自己绑定学员的数据。
- 学员只能查看自己的课表、预约、课时余额、作品等信息。
- 教师只能查看和操作自己授课相关的课表、学员、考勤、作品记录。

### 小程序一期能力

家长/学员端一期建议先做查询和轻量操作：

1. 微信登录与手机号绑定。
2. 我的学员列表。
3. 剩余课时查询。
4. 可约课表查询。
5. 提交约课、取消预约。
6. 查看我的预约记录。
7. 查看作品进度。
8. 提交请假、补课或反馈申请。

教师端可作为一期后半段或二期：

1. 查看今日课表。
2. 查看班级学员。
3. 上课签到和考勤。
4. 维护作品状态。
5. 查看课时量和薪资核算结果。

### 约课与课时余额设计原则

如果支持用户端自主约课，需要补齐课时账户和预约账本，不能只在现有排课表上直接加字段。

建议新增能力：

- 学员课包账户：记录总课时、已消耗课时、冻结课时、剩余课时、有效期。
- 课时流水：记录购买、约课冻结、取消释放、签到扣减、退款冲减、请假返还等动作。
- 预约记录：记录学员、课时、预约来源、预约状态、预约时间、取消时间。
- 可约课表查询：按校区、课程、课包、年龄/等级、容量、时间冲突、有效期过滤。
- 约课事务：校验课时余额、容量、重复预约、时间冲突，通过后创建预约并冻结课时。

推荐课消规则：

- 预约时冻结课时，不直接扣减。
- 取消预约释放冻结课时。
- 签到或上课完成后正式扣减课时。
- 缺勤是否扣课时按机构规则配置。

- 资源预约目前只完成基础 CRUD 和前端下拉选择，后端尚未做教室/设备时间冲突校验。
- 考勤与课消目前完成基础 CRUD，尚未把出勤状态与合同课时余额扣减、补课审批联动成事务流程。

## 全链路业务事件方案

目标：在现有 CRUD 基础上，把关键业务动作收敛为 Service 层事务事件。Controller 只接收请求，Service 负责校验、写入主表、写入关联表、更新状态和金额/课时余额，所有跨表动作使用 `@Transactional(rollbackFor = Exception.class)`。

### 第一阶段：合同与财务闭环

1. 合同创建事件
   - 触发点：新增合同成功。
   - 涉及模块：合同、订单。
   - 处理方案：创建合同时同步生成一笔待支付订单，订单金额取合同应收金额，业务类型标记为报名/课包购买。
   - 事务边界：合同保存、订单保存必须在同一事务内；任一步失败整体回滚。
   - 建议入口：新增 `ContractBizService.createContractWithOrder()`，不要把联动逻辑放在 Controller。

2. 收款成功事件
   - 触发点：收款记录状态变为成功。
   - 涉及模块：收款、订单、合同。
   - 处理方案：保存收款后累加订单已收金额或更新订单支付状态；若订单关联合同，则同步累加合同 `paidAmount`。当合同已收金额大于等于应收金额时，合同可进入生效中或已完成状态，具体规则后续按业务确认。
   - 事务边界：收款保存、订单状态更新、合同已收金额更新必须在同一事务内。
   - 注意点：避免重复回写。建议根据收款状态从非成功变为成功时才触发，已成功记录再次编辑不重复累计。

3. 退款成功事件
   - 触发点：退款记录状态变为已退款。
   - 涉及模块：退款、收款、订单、合同。
   - 处理方案：退款成功后冲减订单/合同已收金额，并按剩余实收金额回退订单支付状态。退款金额不能超过原收款可退余额。
   - 事务边界：退款保存、订单金额冲减、合同金额冲减必须在同一事务内。
   - 注意点：需要定义“可退金额”计算口径，避免多次退款超额。

4. 发票开具事件
   - 触发点：发票状态变为已开票。
   - 涉及模块：发票、订单。
   - 处理方案：发票金额不得超过订单已支付金额或可开票余额；开票成功后可记录订单已开票金额或通过发票汇总计算。
   - 事务边界：发票状态更新和订单开票汇总更新保持一致。

### 第二阶段：考勤与课消闭环

1. 出勤确认事件
   - 触发点：考勤状态变为已出勤/到课。
   - 涉及模块：考勤、课时、合同/课包余额、课消日志。
   - 处理方案：出勤确认后生成课消记录，扣减学员对应合同或课包剩余课时。
   - 事务边界：考勤更新、课消日志写入、课时余额扣减必须在同一事务内。
   - 注意点：需要先明确课时余额存放位置；若当前表结构没有余额字段，需要补字段或新增课包余额表。

2. 请假审批事件
   - 触发点：请假审批通过。
   - 涉及模块：请假、排课、考勤。
   - 处理方案：审批通过后标记该学员课时为请假，不生成课消；必要时释放补课资格。
   - 事务边界：请假状态、考勤状态、补课资格写入保持一致。

3. 补课审批事件
   - 触发点：补课申请审批通过。
   - 涉及模块：补课记录、原课时、目标课时、考勤。
   - 处理方案：审批通过后绑定目标课时，目标课时出勤后再触发课消；原课时不重复扣减。
   - 事务边界：补课审批状态和目标课时绑定保持一致。

### 第三阶段：排课与资源占用

1. 排课创建/更新时间冲突校验
   - 触发点：新增或编辑课时。
   - 涉及模块：排课、教师、教室。
   - 处理方案：同一教师、同一教室在时间区间内不可重复排课。
   - 事务边界：冲突校验和课时保存放在同一 Service 方法内。

2. 资源预约事件
   - 触发点：新增或编辑资源预约。
   - 涉及模块：资源预约、教室、设备、排课。
   - 处理方案：按资源类型校验同一时间段占用；设备预约数量不能超过可用数量。
   - 事务边界：预约保存和设备可用量锁定保持一致。

### 第四阶段：作品流转

1. 上釉记录创建事件
   - 触发点：新增上釉记录。
   - 涉及模块：上釉记录、作品档案。
   - 处理方案：上釉记录保存成功后，作品状态推进为已上釉。
   - 事务边界：记录保存和作品状态更新同事务。

2. 烧制批次完成事件
   - 触发点：烧制批次状态变为已完成。
   - 涉及模块：烧制批次、批次作品明细、作品档案。
   - 处理方案：批次完成后，把批次内作品状态推进为已烧制。
   - 事务边界：批次状态更新、批次明细查询、作品批量更新同事务。

3. 取件记录完成事件
   - 触发点：取件记录状态变为已完成。
   - 涉及模块：取件记录、作品档案。
   - 处理方案：取件完成后，作品状态推进为已取件。
   - 事务边界：取件记录更新和作品状态更新同事务。

4. 损坏记录创建事件
   - 触发点：新增损坏记录。
   - 涉及模块：损坏记录、作品档案。
   - 处理方案：损坏记录保存成功后，作品状态推进为损坏。
   - 事务边界：损坏记录保存和作品状态更新同事务。

### 第五阶段：报表与绩效汇总

1. 每日经营看板汇总事件
   - 触发点：定时任务或手动刷新。
   - 涉及模块：线索、学员、排课、考勤、合同、收款、退款。
   - 处理方案：按校区和日期汇总新增线索、新增学员、转化率、满班率、出勤率、续费率、营收、成本。
   - 事务边界：建议按 `campusId + statDate` 幂等覆盖写入，避免重复累计。

2. 教师薪资核算事件
   - 触发点：按月生成薪资。
   - 涉及模块：教师、排课、考勤、提成规则、薪资记录。
   - 处理方案：按老师和月份汇总课时数量、出勤率、提成金额、扣减金额、应付金额。
   - 事务边界：同一老师同一月份只保留一条核算记录，重复生成应覆盖或拒绝，需业务确认。

### 推荐落地顺序

1. 先做收款成功回写合同/订单，因为它直接影响财务数据准确性。
2. 再做退款冲减，避免收款闭环完成后出现反向金额不一致。
3. 然后做合同创建自动生成订单，减少前台重复录入。
4. 接着做考勤课消，因为它可能需要补余额字段或余额表。
5. 再做作品流转，状态推进规则清晰，适合作为中等风险联动。
6. 最后做报表和绩效汇总，等前面的基础事件稳定后再汇总更可靠。
